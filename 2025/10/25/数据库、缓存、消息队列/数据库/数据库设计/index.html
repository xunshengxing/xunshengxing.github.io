以广告媒体管理系统为例的数据库设计

发表于2025-10-25|更新于2025-10-25|数据库
|总字数:2.1k|阅读时长:6分钟|浏览量:

以下以广告媒体管理系统为例，

广告媒体是指电子屏幕或者纸质展示牌等能够刊登广告的媒介，以下简称媒体

业务需求：

  • 对媒体进行管理：查看所有媒体的广告刊登情况
  • 广告的审批与管理：画面审批、上下刊申请、拍照反馈
  • 运维与故障管理：媒体的报修与维护
  • 销售管理：查看媒体销售状态与合同
  • 用户管理：用户权限与角色管理，微信登录与账密登录
  • 数据统计：广告统计、故障统计
  • ….其他非关键需求

实体

实体能否在现实中独立存在？

虽然传统上实体是现实中可区分的独立对象”，但在数据库建模中，“实体”更多是指“在系统中有独立身份、需要单独管理的数据对象”，它既可以是现实中的独立个体，也可以是为了建模需要而抽象出来的逻辑存在。

现实中的“独立存在” vs 数据库中的“实体”

强实体

这类实体不依赖于其他对象而存在。

  • 学生：有学号、姓名，可以独立存在。

  • 教师、课程、商品、客户

📌特征：有主键，能独立标识自身。

弱实体

这类实体不能独立存在，必须依附于另一个实体。

  • 订单项（Order Item）：没有“订单”就没有“订单项”

  • 宿舍床位：只有在某栋“宿舍楼”存在时才有意义

  • 银行账户交易记录：必须属于某个“账户”

📌 特征：没有完全独立的主键（称为部分键或鉴别符）

实体本质

在数据库设计中，判断一个对象是不是“实体”，关键不是它是否物理独立，而是：

这个对象是否有自己的属性？是否需要被唯一标识？是否参与关系？是否“需要被独立管理和标识

如果答案是“是”，那它就可能是一个实体，哪怕它是依附性的。

比如“学生”和“课程”之间是多对多关系。为了记录“选课成绩”，我们会创建一个叫 选课记录 的实体。

  • 它看起来像是一个“关系”，但因为它有属性（如成绩、选课时间），所以我们把它当作实体处理。

  • 在ER图中，它可以被建模为弱实体或关联实体。

👉 虽然“选课记录”不是一个物理对象，但它在系统中需要被独立存储和查询，因此被视为实体。

实体判断

“存在”不一定是物理的。

  • “合同”不是实物，但它是法律意义上的独立实体。

  • “航班”不是物体，而是事件，但仍可作为实体建模。

  • “项目任务”、“工单”、“日志条目”等都是逻辑存在，但需作为实体管理。

所以，在信息系统中，“存在” = 具有唯一身份 + 可被追踪 + 可被操作

判断标准 说明
是否有唯一标识（主键）？ 是 → 倾向于实体
是否有多个属性需要记录？ 是 → 倾向于实体
是否参与多种关系？ 是 → 更可能是实体
是否可脱离其他对象独立存在？ 可以 → 强实体；不可以 → 弱实体（仍是实体）

所以：实体 ≠ 必须物理独立存在，而是指“在系统中具有独立数据身份的对象”。、

在数据库建模中，“实体”是信息世界的公民——只要它有身份证（主键）、有档案（属性）、能与其他对象互动（关系），它就是合法的“实体”，无论它在现实中是人、物，还是一个事件或关系的实例。

如果你看到一个对象需要被持久化、被查询、被更新，哪怕它是依附性的，也应该考虑将其建模为实体（尤其是弱实体或关联实体）。

E-R图的绘制

识别业务中的实体

实体是需要被独立管理和表示的对象

  • 找重复出现的核心名词

  • 排除属性（如“姓名”不是实体，它是人的属性）

  • 实体通常能拥有自己的属性和参与关系

  1. 媒体（媒体区域、广告刊登情况、媒体ID、媒体套餐、媒体类型、媒体单位、维修记录？、保修记录？）
  2. 广告（广告ID、画面、合同，审批）
  3. 用户（用户ID，账密，微信号？）

属性拆分

属性拆分（尤其是多值、复合、弱实体）应在E-R图设计阶段完成，这是概念建模的核心任务。逻辑模型只是将E-R图规范化为表结构。

媒体拆分

媒体类型、媒体区域，媒体单位独立处理，拆分出来

一个媒体可能归属多个套餐，多值问题，一个属性可能有多个值，进行拆分

维修/保修记录：一个媒体可能有多个维修/保修记录，进行拆分，多值问题

广告刊登情况：这是一个动态状态或行为，不应作为媒体属性，应该单独设置”广告排期“或者”刊登记录”

思考：为什么不能直接设置一个属性：如果有广告刊登就把广告id记录，没有就为空这不是没毛病吗

问题1： 无法记录历史刊登情况，只能记录当前刊登的情况

问题2：无法判断时间是否重叠，

  • 广告A想在2025-11-01~11-30刊登。

  • 广告B也想在2025-11-15~12-15刊登。

只有一个字段无法判断两个广告是否冲突，必需有“开始时间”跟“结束时间”字段，而这只能存放在独立的“刊登记录”

问题3：不支持同一个媒体轮播

  • 某个LED大屏每10秒轮播5个广告。

  • 它“同时”刊登了5个广告。

当前刊登广告ID”字段只能存一个ID，无法表达多广告共存。

必须用一个独立表来记录“哪个广告在哪个媒体上播了多少时间

问题4：无法统计分析（数据丢失）

  • “过去一年哪些媒体最抢手？”

  • “哪个客户投放最多？”

  • “春节期间广告单价上涨了多少？”

所有这些分析都依赖历史排期数据。如果你只存“当前”，这些分析统统做不到。

不要把一个“会发生多次、有时间变化、涉及多个对象”的业务行为，压缩成一个静态字段

  • “广告刊登”是一个行为（广告 → 在 → 媒体 → 上 → 刊登）

  • 它有参与者（广告、媒体）

  • 有时间范围（开始、结束）

  • 有状态变化（待审、已登、结束）

👉 这种复杂的行为，本质上是一个关联事件，应该用一张独立的表来记录，而不是塞进某个实体的一个字段里。

📌 类比：

  • 不能因为“用户买了商品”，就在用户表里加个 当前购买商品ID。

  • 而应该建“订单”表，记录谁、什么时候、买了什么。

拆分后的实体：

媒体：（媒体ID、媒体区域ID、媒体类型ID）

媒体类型：（媒体类型ID、类型名）

媒体单位：（媒体单位ID、单位名）

媒体套餐：（媒体套餐ID，套餐名）

媒体_套餐记录：（媒体套餐ID、媒体ID）

维修记录：（媒体ID，维修记录ID）

保修记录：（保修记录ID，媒体ID）

刊登记录：（刊登ID、刊登状态、刊登时间、媒体ID）

广告拆分

画面：画面是一个复合属性，有多个版本或元数据（尺寸、格式、审核状态），应该设置单独实体

合同：复合属性，

审批：审批是一个动态状态或行为，可能涉及多个环节、审批人、时间、意见。应拆为“审批记录”实体，与“广告”建立一对多关系（一次广告可能多次提交审批）。

拆分后的实体：

广告（广告ID、…）

合同（合同ID、广告ID、客户、…）

画面（图片、尺寸、审批ID）

画面审批记录（审批人、审批时间、画面ID
